hexo deploy问题

1.在家另一台电脑上hexo deploy部署上传博客，遇到一个问题：

fatal: could not read Username for ‘https://github.com': No error

2.找到下面的解决方式：
_config.yml配置文件中将git的配置：

repo: https://github.com/sgven/sgven.github.io.git
修改为repo: git@github.com:sgven/sgven.github.io.git

3.改完之后再次部署 hexo d，还是错误，但是错误已经不同了：

4.原因是没有在 Github 上添加公钥。
使用如下命令测试有没有公钥：

ssh -T git@github.com

如上， Permission denied(publickey)这就表示缺少公钥。

——>缺少公钥，所以要加一个公钥

Github 添加公钥 SSH Keys

本地创建公钥 SSH Keys：

ssh-keygen -t rsa -C "sgvenday@outlook.com"

  • 后面的邮箱即为 github 注册邮箱，也是你登录 Github 的邮箱

  • 之后会要求确认路径和输入密码，一路回车就行。

  • 成功的话会在 默认目录/下生成 .ssh文件夹，进去，打开 id_rsa.pub，复制里面的key即可

Github仓库添加公钥 SSH Keys：

在你的git仓库/Settings/Deploy keys下
打开id_rsa.pub，复制里面的key到github即可

如同所示：

本地可以再测试下公钥有没有添加成功：

ssh -T git@github.com

最后公钥添加成功。可以再次尝试hexo deploy了。

结果如图：成功deploy。

hexo deploy缓存问题

hexo deploy不知道是不是有缓存，几次deploy，线上都没有刷新

解决办法：

干脆一条命令搞定：
hexo clean && hexo g -d --config source/_data/next.yml

hexo资源文件夹功能

对于那些想要更有规律地提供图片和其他资源以及想要将他们的资源分布在各个文章上的人来说，Hexo也提供了更组织化的方式来管理资源。这个稍微有些复杂但是管理资源非常方便的功能可以通过将 config.yml 文件中的 post_asset_folder 选项设为 true 来打开。

_config.yml
post_asset_folder: true //资源文件夹功能开关

当资源文件管理功能打开后，Hexo将会在你每一次通过 

hexo new [layout] <title>

命令创建新文章时自动创建一个文件夹。这个资源文件夹将会有与这个 markdown 文件一样的名字。将所有与你的文章有关的资源放在这个关联文件夹中之后，你可以通过相对路径来引用它们。

理解这段话的意思：

    1. 不仅仅只是自动帮你创建好一样名字的文件夹，方便你统一管理
    1. 要启用了这个开关，才可以通过相对路径引用到它们。——如果没有启用这个开关的话，是引用不到相对路径中的资源的。

遇到的坑

1.这里就遇到一个坑：
使用了新特性，将站点配置和主题部分配置放在同一个配置文件 source/_data/next.yml 中，

我起初觉得这个功能仅仅只是自动创建了一个文件夹，所以就只在根目录下的_config.yml中配置为true了，

我们启动和部署都是用参数–config …/next.yml 来使用自定义的配置的，这个next.yml中没有启用，结果是引用相对路径的图片怎么都看不到。

结论：
简单办法，在根目录下_config.yml和next.yml中都配置为true
——最终目的是：确保运行和发布时使用的配置中,资源文件夹功能开关是启用的

确保配置post_asset_folder: true
即可在运行(hexo s)，发布(hexo g)时，引用到相对路径的图片，从而正确生成插入图片
_posts
|- post1.md
|_ post1
    |- pic1.png

将已经写好的Markdown文档调整为hexo博客

    1. 必须和.md后缀前的名字一致
    1. 而且.md文件和图片文件夹必须都在source/_posts根目录

重点：修改主题配置，使用hexo新特性——Data Files

注意：这个特性是从hexo 3版本引起，需要 Hexo 的版本不低于 3

参考：特性 Benefits

  • 1.为什么要用这个特性？

    原因就是：

    • 我们要对主题做一些个性化处理，启用一些新功能等 都会修改主题配置文件。
    • 所以当我们从git上更新主题，git pull时就可能会冲突，从而导致升级主题失败。

  • 2.这个特性是干嘛的？

    • 通过这个特性，你可以将所有的主题配置放置在站点的 source/_data/next.yml 文件中。
    • 主题配置文件可以不用做任何修改。若后续版本有配置相关的改动时，你仅需在 next.yml 中做相应调整即可——这样就解决了升级和修改的冲突

  • 3.如何使用这个特性？

    • 1.请先确保你所使用的 Hexo 版本在 3 以上

    • 2.在站点的 source/_data 目录下新建 next.yml 文件（_data目录可能需要新建）

    • 3.迁移站点配置文件和主题配置文件中的配置到 next.yml 中

        站点配置文件：即根目录的_config.yml，全部复制进来，
  theme，记得还是指定你用到的主题，
  主题配置文件：也不用全迁移进来，迁移部分要用到的配置即可，做个性化配置处理。
  
  我认为，这里是theme:指定了引用的主题配置，
  然后next.yml中的所有配置会合并主题配置且覆盖键值重名的配置，
  这样就把需要改的主题配置复制到next.yml中，再个性化修改
  相当于assign(主题配置,next.yml)

    • 4.使用 –config source/_data/next.yml 参数启动服务器, 生成或者部署。

      例如:

    • hexo clean && hexo s –config source/_data/next.yml

    • hexo clean –config source/_data/next.yml && hexo g –config source/_data/next.yml

  • 3.如果仅仅只是打开开关，部署之后会发现文章的【字数统计】和【阅读时长】后面没有对应的xxx字，xx分钟等字样，只有光秃秃的数字在那里。

解决方案：

​找到Blog\themes\next\layout_macro\post.swig 文件，将“字”、“分钟” 字样添加到如下位置即可：

<span title="{{ __('post.wordcount') }}">
     {{ wordcount(post.content) }} 字
</span>

    ...
<span title="{{ __('post.min2read') }}">
     {{ min2read(post.content) }} 分钟
</span>

然后可以看到显示如：阅读时长 ≈ 2 分钟 ,但比如不需要 ≈ 显示，修改这个地方：

{% if theme.post_wordcount.item_text %}
	        {{ __('post.min2read') }} ≈
	 {% endif %}

把这里的≈删除即可。（PS：这个是 html 的特殊字符的表示）

hexo入门概述

  • 学习hexo的目的：

    使用hexo是要搭建博客——>在这之前，先了解一下使用github搭建个人域名的网站，用来发布我们的博客。
  • 了解一下，入门过程：
    • 先了解一下hexo是什么
    • 安装hexo
    • 初始化hexo的建站环境
    • 运行hexo server来启动，使用hexo的服务。
      • hexo会解析根目录下的source文件夹中的.md Markdown文件，生成html静态文件
      • 打开http://localhost:4000,即可看到效果，
    • 最后，发布到线上的静态文件效果就是本地看到的。

hexo是什么：

- Hexo 是一个快速、简洁且高效的博客框架
- Hexo 使用 Markdown 解析文章，在几秒内，即可利用靓丽的主题生成静态网页。

安装：

参考官网 安装 

先安装好 Node.js 和 Git

运行命令：$ npm install -g hexo-cli

建站

参考官网 初始化 建站 环境。

运行命令：$ hexo init <folder>——Hexo 将会在指定文件夹中新建所需要的文件

如 hexo init blog

启动hexo,看运行效果

cd <folder>——进入到hexo建站环境根目录
hexo server(简写hexo s)

#运行hexo，指定配置
hexo clean && hexo s --config source/_data/next.yml

如：

cd blog

hexo server

浏览器打开：http://localhost:4000/

然后可以写博客，把你的.md笔记放入source文件夹，做些调整，发布了。

  • （当然想要更好看，可以更改主题，加上需要的主题配置，在我下一篇博客可以学习下）

  • （发布时，可以使用hexo的一键部署功能，改一下根目录_config.yml配置中的git,将博客发布到github仓库）

    • 如果遇到这个错误，ERROR Local hexo not found，解决办法如下：
    • 更新 Nodejs 到最新版本。（我就是node版本过低，一直报这个错）

hexo发布部署

命令

Hexo 提供了快速方便的一键部署功能，让您只需一条命令就能将网站部署到服务器上。
$ hexo deploy

当然，在发布之前，除了配置文件，也需要先生成静态文件（即要发布的内容）

$ hexo generate

不过，这两条命令也可以放一起写，变成一条命令：

$ hexo generate --deploy
$ hexo deploy --generate
两个命令都可以，作用是相同的
简写：
    $ hexo g -d
    $ hexo d -g

配置

命令我们就用合并的简写，现在关心下配置：
因为我们使用git仓库：

  • 1.安装 hexo-deployer-git。（自动部署发布工具）

      $ npm install hexo-deployer-git --save

  • 2.修改配置。

      deploy:
    type: git
    repo: <repository url>
    branch: [branch]
    # message: [message],这里就用默认的
    • message：自定义提交信息 (默认为 Site updated: { now(‘YYYY-MM-DD HH:mm:ss’) })——（坑，官网说明是连着的两个左右花括号{}，在hexo中解析失败，花括号“{”,”}”不能紧挨着重复用）
