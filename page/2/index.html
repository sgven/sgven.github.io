# 添加父镜像，如java的运行时jdk

# Run app.jar when the container launches
CMD ["java","-jar","./app/app.jar"]

虚拟化技术与容器技术

误解：

  • 1.实现虚拟化的程序本身就要占用计算机的资源，应用程序本身也不会降低对资源的消耗，怎么又会产生 1 + 1 < 2 的效果呢？

    注意：这里是提高计算机资源使用率，而非减少程序资源的占用率。这两者不是同一个概念，虚拟化技术能够提高计算机资源的使用率，将闲置资源拿出来，分享给另外一些程序，让计算机资源不被浪费。

    例如，有一台运行 Nginx 的机器，因为Nginx对系统资源的消耗并不高，让系统 95% 以上的资源处于闲置状态。这时可以通过虚拟化技术，让其他程序也部署到这台机器上来，充分利用闲置的资源。好处是不需要再为这些程序单独部署机器，从而节约不少的成本。

  • 2.本身就可以在操作系统里安装这些程序并且同时运行，为什么还要把它们分别装到不同的虚拟环境中去呢？

    虽然我们能够在操作系统里同时运行多个程序，但前提得是这些程序本身不存在冲突。虚拟化技术通过资源隔离的方式，无形地也可以把这些程序隔离在不同的虚拟环境中，程序就不会互相干扰或争抢资源了。

  • 3.docker容器实际上在宿主机(操作系统)看来，其实就是一个进程，直接就运行在宿主机上的。既不转换指令，也没有解决程序跨平台兼容的问题，这算哪门子虚拟化技术。

    • docker镜像就相当于(静态的)程序

    • docker容器就相当于(运行中的)进程

    • 正是这种原因，很多人并不认同容器技术属于虚拟化技术的范畴。

    • 不过，另一派观点认为，容器技术提供了相对独立的应用程序运行的环境，也提供了资源控制的功能，所以我们依然可以归纳其为一种实现不完全的虚拟化技术。——>可以认为是一种操作系统虚拟化的范畴，需要操作系统提供虚拟化的支持。

虚拟机 vs 容器技术

虚拟机

虚拟机的一个致命的缺陷，就是所有的指令都必须经过虚拟机监视器的处理。导致性能低下。

容器技术

容器技术是一种全新意义上的虚拟化技术，其应该属于操作系统虚拟化的范畴，也就是在由操作系统提供虚拟化的支持。

容器技术近年来已经是一个火遍大江南北的概念了，它能够兴起的很重要的一个原因是其在运行性能上要远超虚拟机等其他虚拟化实现。

为什么容器技术能够造就近乎完美的运行效率呢？

因为容器技术压根没有做指令转换。是的，有时候解决问题的最佳方法就是不解决它。

没有了虚拟操作系统和虚拟机监视器这两个层次，大幅减少了应用程序运行带来的额外消耗。

更准确的来说，所有在容器中的应用程序其实完全运行在了宿主操作系统中，与其他真实运行在其中的应用程序在指令运行层面是完全没有任何区别的。

Docker Engine和Docker Machine有什么区别？

Docker Engine

当人们说“Docker”时，他们通常指的是Docker Engine，Docker Engine是由Docker守护程序组成的客户端 - 服务器应用程序。包括以下主要组件：

  • Docker守护程序
  • 指定用于与守护进程交互的接口的REST API，
  • 以及与守护进程通信的命令行界面（CLI）客户端


Docker Machine

您可以使用Machine在一个或多个虚拟系统上安装Docker Engine，Docker Machine是一个用于配置和管理的工具。

Docker 底层组成

  • 1.Namespace：隔离技术的第一层，确保 Docker 容器内的进程看不到也影响不到 Docker 外部的进程。
  • 2.Control Groups：LXC 技术的关键组件，用于进行运行时的资源限制。
  • 3.UnionFS（文件系统）：容器的构件块，创建抽象层，从而实现 Docker 的轻量级和运行快速的特性。

Docker 组件/元素

Docker 有基本组件和元素各三个，视频 Docker Onboarding(自带梯子)，可以让你快速弄清三者之间的关系。

  • 组件：
    • Docker Client：用户和 Docker 守护进程进行通信的接口，即docker 命令。
    • Docker Daemon(守护进程)：宿主机上用于用户应答用户请求的服务。
    • Docker Registry：用户进行私有、公有 Docker 容器镜像托管，即Docker 仓库。
  • 元素：
    • Docker Container：用于运行应用程序的容器，包含操作系统、用户文件和元数据。
    • Docker Image：只读的 Docker 容器模板，简言之就是系统镜像文件。
    • DockerFile：进行镜像创建的指令文件。

docker镜像(image)

Docker 镜像是 Docker 容器运行时的只读模板，每一个镜像由一系列的层 (layers) 组成。Docker 使用 UnionFS 来将这些层联合到单独的镜像中。

因为Docker镜像层都有唯一的 64 长度的 Hash 码（编码），我们就能够区分不同的镜像层并能保证它们的内容与编码是一致的。这带来了另一项好处，就是允许我们在镜像之间共享镜像层。

镜像可以共用一些存储空间，达到 1 + 1 < 2 的效果，为我们在同一台机器里存放众多镜像提供了可能。正是因为这种分层设计，Docker 是如此的轻量。

docker容器(container)

可怜的容器，被Namespace欺骗，又被Cgroups限制，在这样的环境下，还发挥着自己生命的意义-与应用程序 同生共死。

镜像与容器的关系

两者的关系可以类比为：
docker镜像(image)相当于java类
docker容器(container)相当于java类创建的对象(也就是镜像的实例)

从操作系统来看：
docker镜像就相当于(静态的)程序
docker容器就相当于(运行中的)进程

数据卷(volume)

数据卷的生存周期独立于容器，容器消亡，数据卷不会消亡。

为什么docker能强势崛起

docker发展历史

2013年，曾经被人们寄予厚望的云计算技术，也已经从当初虚无缥缈的概念蜕变成了成了实实在在的虚拟机和账单。那时主流的是以Cloud Foundry为代表的PasS开源项目，Cloud Foundry 项目已经基本度过了最艰难的概念普及和用户教育阶段，吸引了大批厂商，开启了以开源 PaaS 为核心构建平台层服务能力的变革。

（当时叫dotCloud公司）Docker 公司，也是这股 PaaS 热潮中的一份子。而它的主打产品由于跟主流的 Cloud Foundry 社区脱节，长期以来也无人问津。眼看就要被如火如荼的 PaaS 风潮抛弃，dotCloud 公司却做出了这样一个决定：开源自己的容器项目 Docker。

这个决定在当时根本没人在乎。 “容器”这个概念从来就不是什么新鲜的东西，也不是 Docker 公司发明的。即使在当时最热门的 PaaS 项目 Cloud Foundry 中，容器也只是其最底层、最没人关注的那一部分。以当时的事实标准 Cloud Foundry 为例，先来解说一下PasS技术。

PaaS 项目被大家接纳的一个主要原因，就是它提供了一种名叫“应用托管”的能力。当时主流做法是，租一批虚拟机，像管理物理服务器那样，用脚本或者手工的方式在这些机器上部署应用。

这个部署过程难免会碰到云端虚拟机和本地环境不一致的问题，所以当时的云计算服务，比的就是谁能更好地模拟本地服务器环境，能带来更好的“上云”体验。而 PaaS 开源项目的出现，就是当时解决这个问题的一个最佳方案。

举个例子，虚拟机创建好之后，运维只需要在这些机器上部署一个 Cloud Foundry 项目，然后开发者只要执行一条命令就能把本地的应用部署到云上：

cf push " 我的应用 "

Cloud Foundry 项目把应用和启动脚本打进一个压缩包内，然后运行在容器中。

而 Docker 项目，实际上跟 Cloud Foundry没什么太大的不同。然而，短短几个月，Docker 项目就迅速崛起了。以至于很多PaaS 社区还没来得及成为它的竞争对手，就直接被宣告出局了。

它们两个容器在大部分功能和实现原理上都是一样的。不一样的法宝就是，Docker 镜像。

PaaS 提供了一个打包功能能够帮助用户大规模部署应用到集群里。但这个却是它的软肋。问题的根本原因是，这个打包过程，没有任何章法可循，明明在本地运行得好好的应用，却需要做很多修改和配置工作才能在PaaS 里运行起来。而这些修改和配置，并没有什么经验可以借鉴，基本上得靠不断试错。

而Docker 镜像解决的，恰恰就是打包这个根本性的问题。这个镜像，其实就是一个压缩包。大多数Docker镜像是由一个完整操作系统的所有文件和目录构成的，所以这个压缩包里的内容跟你本地开发用的操作系统是完全一样的。

在这个过程中，你完全不需要进行任何配置或者修改，因为这个压缩包赋予了你一种极其宝贵的能力：本地环境和云端环境的高度一致！

这，正是 Docker 镜像的精髓。

所以Docker是解决了PasS的根本性问题，Docker提供了一种非常便利的打包机制：直接打包了应用运行所需要的整个操作系统，从而保证了本地环境和云端环境的高度一致，避免了用户通过“试错”来匹配两种不同运行之间差异的痛苦过程。

docker兴起原因(总结)

  • 容器技术的兴起源于 PaaS 技术的普及；
  • Docker 项目通过“容器镜像”，解决了应用打包这个根本性难题。
  • Docker开源，和开发者有一种与生俱来的亲密感。
  • 还有一点原因是：它的运行性能要远超虚拟机等其他虚拟化实现。(在下一篇介绍相关虚拟化、容器等技术概念时会具体讲到)

YAML

YAML是JSON的超集，是用于指定分层配置数据的便捷格式。比json更轻量易读的数据序列化表示语言，YAML使用冒号加缩的方式表示层级关系，在多层嵌套时有更好的可读性。

官网：

  • YAML: YAML Ain’t Markup Language，YAML不是标记语言。
  • What It Is：YAML是一个针对所有编程语言的人类友好的数据序列化标准。

SpringBoot为什么使用YAML

  • XML：在数据格式描述和较复杂数据内容展示方面，比较优秀，但是XML的配置十分繁琐；
  • properties：在表现层级关系和结构关系的时候，十分欠缺；
  • JSON：JSON格式比XML格式，更加方便，所以很多配置文件都习惯使用JSON；
  • YAML：比JSON更轻量易读的数据序列化表示语言

YAML语法

基本要求

  • 1.YAML大小写敏感；
  • 2.使用缩进代表层级关系；
  • 3.缩进：YAML推荐使用空格作为缩进，避免了在不同编辑器中对tab的表示形式不同而可能产生误解，不要求空格个数，只需要相同层级左对齐（一般2个或4个空格）

注释

和properties相同，使用#作为注释，不过，YAML中只有行注释。

文档(document)

  • 一个YAML文件中可以保存多个文档；
  • 文档之间以“—”作为间隔，以“…”作为结束标志。

暂时把一个doucment视为一个object序列化后得到的yaml配置信息

# documents example
---
site: {name: sina, url: http://www.sina.com.cn}
---
site: {name: google, url: http://www.google.com}

数据结构

YAML的设计者认为在配置文件中所要表达的数据内容有三种类型:

  • 标量(Scalar，如字符串和整数等)
  • 序列（Sequence，如数组）
  • Mapping（类似hash的key/value pair）

特殊符号

  • 1.---: 同一个YAML文件中，文档之间以“—”作为间隔，用—表示一个文档的开始；

  • 2....： 和 --- 配合使用，代表一个文件的结束；

      ---
  xx:a
  ...
  ---
  xx:b
  ...

    相当于在一个yaml文件中连续写了两个yaml配置项

  • 3.!!： 使用!!做类型强行转换，允许转型的类型很多。比如：

      # !!str强转为字符串
  string:
  - !!str 54321
  - !!str true

  # !!set，下面是将数组解析为set，重复的Sammy Sosa去掉
  --- !!set
  - Mark McGwire: 65
  - Sammy Sosa: 63
  - Sammy Sosa: 63

  • 4.> 在字符串中折叠换行 和 | 保留换行符 ：

      accomplishment: >
   Mark set a major league
   home run record in 1998.
  stats: |
   65 Home Runs
   0.278 Batting Average

  结果：
  # 将换行符转化成了空格，注意：每行的文本前一定要有一个空格
  accomplishment=Mark set a major league home run record in 1998.

  # | 符号保留了换行符
  stats=65 Home Runs
    0.278 Batting Average,

  • 5.引用：重复的内容在YAML中可以使用&来完成锚点定义，使用*来完成锚点引用：

      hr:
  - Mark McGwire
  # 设置了一个锚点（引用），名称是SS
  - &SS Sammy Sosa
  rbi:
  # 引用了锚点*SS
  - *SS 
  - Ken Griffey
  
  结果为：
  {rbi=[Sammy Sosa, Ken Griffey], hr=[Mark McGwire, Sammy Sosa]}

    • 也可以这样定义：

        SS: &SS Sammy Sosa
  hr:
   - Mark McGwire
   - *SS
  rbi:
   - *SS 
   - Ken Griffey

      注意，不能独立的定义锚点，比如直接这样写：&SS Sammy Sosa
      hr

        &SS Sammy Sosa
  hr：
   - Mark McGwire
   - *SS

    • 另外，锚点能够定义更复杂的内容，比如：

        default: &default
  - Mark McGwire
  - Sammy Sosa

maven依赖版本统一管理

maven依赖版本管理，主要是避免多模块间依赖的版本冲突，保持依赖版本的统一。

1.常规做法：

  • 在parent model(父pom)中，使用dependencyManagement预定义所有模块需要用到的dependency(依赖)的版本

  • 然后，子model按需引入parent中预定义的依赖。

  • 这种做法的坏处是：

maven的单继承：maven的继承跟java一样，单继承，也就是说子model中只能出现一个parent标签。

这样parent模块(父pom)中，dependencyManagement中预定义太多的依赖，造成pom文件过长，而且很乱；

——》如何让这些依赖可以分类并清晰的管理？

2.dependencyManagement中使用import scope来实现多继承

这种做法是：将原来一个父pom中的dependencyManagement版本管理分散到多个pom来维护依赖版本。然后通过在dependencyManagement中使用import scope来实现多继承。

如何使用：

  • 1、maven2.9以上版本
  • 2、将dependency分类，每一类建立单独的pom文件
  • 3、在需要使用到这些依赖的子model中，使用dependencyManagement管理依赖，并import scope依赖
  • 注意：scope=import只能用在dependencyManagement里面,且仅用于type=pom的dependency

如下：

<dependencyManagement>
        <dependencies>
        <dependency>
            <!-- Import dependency management from Spring Boot -->
            <groupId>org.springframework.boot</groupId>
            <artifactId>spring-boot-dependencies</artifactId>
            <version>2.0.5.RELEASE</version>
            <type>pom</type>
            <scope>import</scope>
        </dependency>
    </dependencies>
</dependencyManagement>

redis简介

Redis 是一种键值（Key-Value）数据库，相对于关系型数据库（比如 MySQL），Redis 也被叫作非关系型数据库。

Redis 中只包含“键”和“值”两部分，只能通过“键”来查询“值”。正是因为这样简单的存储结构，也让 Redis 的读写效率非常高。

redis通常用于缓存。Redis 主要是作为内存数据库来使用，也就是说，数据是存储在内存中的。尽管它经常被用作内存数据库，但是，它也支持将数据存储在硬盘中。

redis使用及相关概念

1.启动redis服务(指定配置文件redis.windows.conf):

redis-server.exe redis.windows.conf

2.使用redis(图形)客户端连接使用redis服务。

3.Redis数据库的概念

注意：Redis支持多个数据库，并且每个数据库的数据是隔离的不能共享，并且基于单机才有，如果是集群就没有数据库的概念。

  • 1 配置文件中，可以修改数据库database大小
    （默认16，无上限，数据库对外都是从0开始的递增数字命名，0,1,2,…）

  • 2.默认启动redis是数据库0，切换数据库命令

    切换数据库，比如运行命令：

      redis 0>: select 1
  "OK"
  redis 1>:

这些以数字命名的数据库又与我们理解的数据库有所区别

1.首先Redis不支持自定义数据库的名字，每个数据库都以编号命名，开发者必须自己记录哪些数据库存储了哪些数据；

2.Redis也不支持为每个数据库设置不同的访问密码，所以一个客户端要么可以访问全部数据库，要么连一个数据库也没有权限访问；

3.最重要的一点是多个数据库之间并不是完全隔离的，比如FLUSHALL命令可以清空一个Redis实例中所有数据库中的数据。

综上，这些数据库更像是一种命名空间，而不适宜存储不同应用程序的数据。不同的应用应该使用不同的Redis实例存储数据。

由于Redis非常轻量级，一个空Redis实例占用的内在只有1M左右，所以不用担心多个Redis实例会额外占用很多内存。

4.Redis HyperLogLog

  • Redis 在 2.8.9 版本添加了 HyperLogLog 结构。
  • Redis HyperLogLog 是用来做基数统计的算法，HyperLogLog 的优点是，在输入元素的数量或者体积非常非常大时，计算基数所需的空间总是固定 的、并且是很小的。
  • 在 Redis 里面，每个 HyperLogLog 键只需要花费 12 KB 内存，就可以计算接近 2^64 个不同元素的基 数。
  • 但是，因为 HyperLogLog 只会根据输入元素来计算基数，而不会储存输入元素本身，所以 HyperLogLog 不能像集合那样，返回输入的各个元素。

什么是基数?

比如数据集 {1, 3, 5, 7, 5, 7, 8}， 那么这个数据集的基数集为 {1, 3, 5 ,7, 8}, 基数(不重复元素)为5。基数估计就是在误差可接受的范围内，快速计算基数。

将多个 HyperLogLog 合并为一个 HyperLogLog:

6.Redis 事务：

单个 Redis 命令的执行是原子性的，但 Redis 没有在事务上增加任何维持原子性的机制，所以 Redis 事务的执行并不是原子性的。

事务可以理解为一个打包的批量执行脚本，但批量指令并非原子化的操作，中间某条指令的失败不会导致前面已做指令的回滚，也不会造成后续的指令不做。

7.Redis 数据备份与恢复：

数据备份：

  • SAVE命令将在 redis 安装目录中创建dump.rdb文件

    我们这里备份文件的路径，同配置文件(redis.windows.conf)的路径一样C:\Users\Administrator，该文件被创建在这个目录下，是设置了环境变量的缘故

  • BGSAVE命令在后台执行，也是创建 redis 备份文件

恢复数据：

如果需要恢复数据，只需将备份文件 (dump.rdb) 移动到 redis 安装目录并启动服务即可。获取 redis 目录可以使用 CONFIG 命令，如下所示：

0>CONFIG GET dir
 1)  "dir"
 2)  "C:\Users\Administrator"

（同样设置了环境变量，我们这里SAVE后就不需要移动了，或者是其他的备份(dump.rdb)移动到这个目录下 C:\Users\Administrato）

8.Redis 管道技术：

Redis是一种基于客户端-服务端模型以及请求/响应协议的TCP服务。通常情况下一个请求会遵循以下步骤：

  • 客户端向服务端发送一个查询请求，并监听Socket返回，通常是以阻塞模式，等待服务端响应。
  • 服务端处理命令，并将结果返回给客户端。

Redis 管道技术
Redis 管道技术可以在服务端未响应时，客户端可以继续向服务端发送请求，并最终一次性读取所有服务端的响应。

管道技术的优势
管道技术最显著的优势是提高了 redis 服务的性能。
